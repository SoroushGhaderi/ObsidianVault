Lesson 0001: JWT Extraction from Authorization Header

Topic: JWT Gateway Auth  |  Prerequisites: Python basics, HTTP headers
Goal: Extract a numeric user_id from a Bearer JWT token using the gateway-trust pattern.

The Problem

Your API gateway validates JWTs and forwards requests to your backend service. The backend needs the user's identity but should not re-verify the token. How do you extract a stable user identifier from the request?

Architecture: The Gateway-Trust Pattern

Client Gateway Backend Service │ │ │ │── POST /api/chat ──────▶│ │ │ Authorization: Bearer │ │ │ eyJhbGciOi... │ │ │ │── Validate JWT ────────▶│ (internal) │ │ signature, expiry, │ │ │ audience, issuer │ │ │ │ │ │── Forward request ─────▶│ │ │ (original JWT intact) │ │ │ │ │ │ Decode payload │ │ Extract UserId │ │ Decode HashId → numeric │ │ │ │◀── Response ───────────│◀── Response ─────────────│

Minimum Concepts

1. JWT Structure

A JWT is three Base64url-encoded parts separated by dots:

eyJhbGciOiJIUzI1NiJ9.eyJVc2VySWQiOiI3NzEzZ2tuIn0.signature_here
│   header            │  payload                        │  sig

The payload contains claims. In our case, UserId is a HashId string like "7713gkn".

2. Bearer Token Extraction

The Authorization header format is Bearer <token>. You must split and validate the prefix:

authorization = request.headers.get("authorization")
# "Bearer eyJhbGci..." → ["Bearer", "eyJhbGci..."]
parts = authorization.split(" ", 1)
token = parts[1]  # the raw JWT

3. Decoding Without Verification

Since the gateway already verified the signature, you can decode with signature verification disabled. Keep any lightweight claim checks that still help with correctness:

import jwt

payload = jwt.decode(
    token,
    options={
        "verify_signature": False,
        "verify_exp": False,
        "verify_aud": False,
        "verify_iss": False,
    },
    algorithms=["HS256", "RS256"],
)
# payload = {"UserId": "7713gkn", "Type": "Customer", ...}

4. HashId Decoding

The UserId claim is a HashId — a reversible encoding of a numeric ID:

from hashids import Hashids

hashids = Hashids(
    salt="your-salt-here",
    min_length=6,
    alphabet="abdfghklmnoprstwxyz123456789",
)

user_hash = payload["UserId"]  # "7713gkn"
decoded = hashids.decode(user_hash)  # (10652,)
user_id = str(decoded[0])  # "10652"

Worked Example: Complete FastAPI Integration

# app/core/jwt_auth.py
from fastapi import HTTPException, Request, status
import jwt
from hashids import Hashids
from app.core.config import settings

_hashids = Hashids(
    salt=settings.HASHIDS_SALT,
    min_length=settings.HASHIDS_MIN_LENGTH,
    alphabet=settings.HASHIDS_ALPHABET,
)

def extract_user_id_from_request(request: Request) -> str:
    # 1. Extract Bearer token
    auth = request.headers.get("authorization")
    if not auth:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Missing Authorization header.")
    parts = auth.split(" ", 1)
    if len(parts) != 2 or parts[0].lower() != "bearer":
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Invalid Authorization format.")

    # 2. Decode JWT (no signature check — gateway handled it)
    try:
        payload = jwt.decode(
            parts[1],
            options={"verify_signature": False, "verify_exp": False,
                     "verify_aud": False, "verify_iss": False},
            algorithms=["HS256", "RS256"],
        )
    except jwt.PyJWTError as exc:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Invalid JWT token.") from exc

    # 3. Extract and decode HashId
    hash_id = payload.get("UserId")
    if not hash_id:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="UserId claim not found.")

    decoded = _hashids.decode(hash_id)
    if not decoded:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Invalid UserId in token.")

    return str(decoded[0])
# app/api/v1/endpoints/chat.py
from fastapi import Depends, Request
from app.core.jwt_auth import extract_user_id_from_request

async def get_current_user_id(request: Request) -> str:
    return extract_user_id_from_request(request)

@router.post("/message")
async def handle_message(
    request: MessageRequest,
    user_id: str = Depends(get_current_user_id),
):
    # user_id = "10652" — ready to use

Retrieval Exercise

Exercise 1: Trace the Data Flow

Given this JWT payload:

{"UserId": "7713gkn", "Type": "Customer", "exp": 1782555915}

And this HashIds config:

salt = "ecawiasqrpqrgyhwnolrudpbsrwaynbqdayndnmcehjnwqyouikpodzaqxivwkconwqbhrmxfgccbxbyljguwlxhdlcvxlutbnwjlgpfhjgqbegtbxbvwnacyqnltrby"
min_length = 6
alphabet = "abdfghklmnoprstwxyz123456789"

What numeric user_id does the backend use? What happens if the salt is wrong?

user_id = 10652

If the salt is wrong, hashids.decode("7713gkn") returns an empty tuple, and the backend raises 401: Invalid UserId in token. The HashId must be decoded with the exact same salt, min_length, and alphabet used to encode it.

Exercise 2: Security Analysis

This pattern skips JWT signature verification. Answer:

  1. What attacker could exploit this?
  2. What prevents the attack?
  3. When would this pattern be unsafe?
  1. An attacker with network access could craft a JWT with any UserId and send it directly to the backend, impersonating any customer.
  2. Network-level controls: the backend only accepts traffic from the gateway (VPC rules, service mesh, firewall). The attacker can't reach the backend directly.
  3. Unsafe when: the backend is publicly reachable, no network isolation exists, or the gateway doesn't validate tokens (double-submit pattern without verification).

Exercise 3: Design Decision

Your team is building a new microservice. The gateway validates JWTs. Should the service:

  • A: Skip JWT verification (gateway-trust pattern)
  • B: Verify JWT signature with shared JWKS endpoint
  • C: Verify JWT signature with hardcoded secret

Rank by: development speed, security, operational complexity. Which do you choose and why?

A (Gateway-trust): Fastest dev, lowest ops complexity, acceptable security for internal services behind gateway.

B (JWKS verification): Highest security, moderate ops (must manage JWKS endpoint), slower dev. Best for zero-trust or public-facing.

C (Hardcoded secret): Avoid. Secret rotation requires redeployment. Worse than A or B in every dimension.

Recommendation: A for internal services behind gateway. B if moving to zero-trust. Never C.

Key Takeaways

  • The Authorization: Bearer <token> header carries the JWT.
  • Gateway-trust pattern: decode JWT without signature verification.
  • HashIds are reversible encodings, not encryption — anyone with the salt can decode.
  • FastAPI Depends() centralizes auth logic and makes testing easy.
  • Security relies on network boundary, not cryptographic verification.
  • Keep the backend dependency small, explicit, and easy to test.
