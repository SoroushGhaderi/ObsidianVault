Lesson 0002: Claim Validation and Auth Dependency Design

Topic: JWT Gateway Auth | Prerequisites: Lesson 0001, FastAPI dependencies, JWT basics
Goal: Make the backend auth dependency small, explicit, and safe by validating only what still matters after gateway verification.

Why This Lesson Exists

Lesson 0001 covered the happy path: the gateway validates the token, the backend extracts the identity, and HashIds convert the claim into a numeric user ID. The next question is more professional: what should the backend still validate, and what should it deliberately ignore?

The Design Principle

The backend should not reimplement gateway authentication, but it should still defend itself against malformed input, contract drift, and accidental misuse.

Rule of thumb Validate structure, required claims, and data shape. Do not turn the backend into a second auth gateway unless you actually need independent trust.

Auth Flow

Client request Sends Authorization: Bearer <jwt> to the API.
↓
Gateway Verifies signature, issuer, audience, and expiry. This is the trust boundary.
↓
Backend auth dependency Checks header shape, decodes payload, requires UserId, decodes HashId, fails closed.
↓
Endpoint Receives a clean user_id string and focuses on business logic only.
Client → Gateway → Backend auth dependency → Endpoint (token) (verify) (validate shape) (business logic)

What the Backend Should Check

1. Authorization Header Shape

The handler should reject missing or malformed bearer tokens immediately.

def extract_bearer_token(authorization: str | None) -> str:
    if not authorization:
        raise HTTPException(status_code=401, detail="Missing Authorization header.")

    scheme, _, token = authorization.partition(" ")
    if scheme.lower() != "bearer" or not token:
        raise HTTPException(status_code=401, detail="Invalid Authorization header.")

    return token

2. Payload Decoding Without Signature Verification

The backend can decode the payload to read claims, but it should not pretend this is full token verification.

def decode_unverified_jwt(token: str) -> dict[str, object]:
    try:
        return jwt.decode(
            token,
            options={
                "verify_signature": False,
                "verify_exp": False,
                "verify_aud": False,
                "verify_iss": False,
            },
            algorithms=["HS256", "RS256"],
        )
    except jwt.PyJWTError as exc:
        raise HTTPException(status_code=401, detail="Invalid JWT token.") from exc

3. Required Claim Presence

The backend should require the identity claim it depends on. If that claim is missing, the request is invalid.

def get_required_claim(payload: dict[str, object], claim_name: str) -> str:
    value = payload.get(claim_name)
    if not isinstance(value, str) or not value.strip():
        raise HTTPException(status_code=401, detail=f"Missing or invalid {claim_name} claim.")
    return value

4. HashId Decoding

The encoded user identity is a transport format, not a privilege boundary. Decode it carefully and fail closed.

def decode_user_id(hash_id: str) -> str:
    decoded = _hashids.decode(hash_id)
    if not decoded:
        raise HTTPException(status_code=401, detail="Invalid UserId in token.")
    return str(decoded[0])

A Clean Dependency

async def get_current_user_id(request: Request) -> str:
    token = extract_bearer_token(request.headers.get("authorization"))
    payload = decode_unverified_jwt(token)
    user_hash = get_required_claim(payload, "UserId")
    return decode_user_id(user_hash)

This is usually the right shape for internal service auth:

  • One dependency owns the whole extraction path.
  • Endpoints depend on a plain `user_id` value.
  • Tests can override a single function instead of patching many internals.

What Not To Do

Anti-pattern Do not spread auth logic across multiple route handlers, and do not silently accept a missing claim just because the gateway usually sets it. 
# Bad: duplicated logic and weak failure handling
auth = request.headers.get("authorization")
if auth:
    token = auth.split(" ")[1]
    payload = jwt.decode(token, options={"verify_signature": False})
    user_id = payload.get("UserId", "0")  # dangerous default

The dangerous part above is not just the missing signature check. It is the fallback to a default identity. In auth code, defaults can become impersonation bugs.

Testing Strategy

A good auth dependency is easy to test at the boundary.

def test_message_endpoint_with_authenticated_user(client, app):
    from app.core.jwt_auth import get_current_user_id

    app.dependency_overrides[get_current_user_id] = lambda: "10652"
    try:
        response = client.post("/message", json={"text": "hello"})
        assert response.status_code == 200
    finally:
        app.dependency_overrides.pop(get_current_user_id, None)

That pattern keeps the test focused on business behavior instead of JWT plumbing.

Failure Modes to Watch

  • Missing Authorization header.
  • Non-bearer tokens or malformed header values.
  • JWT payload missing the expected identity claim.
  • HashId configuration drift across services.
  • Accidental defaults that turn auth failures into valid identities.

Retrieval Exercise

Exercise 1: Decide What the Backend Owns

For each item, decide whether the backend should validate it, trust it from the gateway, or ignore it:

  • Authorization header exists
  • JWT signature
  • UserId claim exists
  • UserId decodes to a number
  • exp claim exists
  • Authorization header exists: validate.
  • JWT signature: trust the gateway in this pattern.
  • UserId claim exists: validate.
  • UserId decodes to a number: validate.
  • exp claim exists: usually trust the gateway, but backend may inspect it for defense in depth if required by policy.

Exercise 2: Spot the Bug

Why is this code dangerous?

user_id = payload.get("UserId", "000000")
return str(_hashids.decode(user_id)[0])

It silently accepts a missing claim and substitutes a fallback identity. If the request is malformed or malicious, the code may authenticate the wrong user instead of failing closed.

Key Takeaways

  • Gateway-trust removes duplicate signature verification, not all validation.
  • The backend should validate header shape, claim presence, and data shape.
  • Defaults in auth code are risky because they can become silent impersonation paths.
  • One dependency should own the auth path from header to user ID.
  • Tests should override the dependency, not re-create JWT mechanics in every test.
