Lesson 2: Event-Driven Backend for Scalable Chatbots

Goal: Understand how event-driven architecture enables scalable, real-time chatbot systems.

Concept Explanation

Traditional request-response architectures block resources while waiting for LLM responses (which can take seconds). Event-driven architecture (EDA) decouples components, allowing asynchronous processing.

Core patterns:

  1. Message Queues: Kafka, RabbitMQ, SQS for reliable message delivery.
  2. Event Streaming: Real-time token-by-token response delivery.
  3. CQRS: Separate read/write models for chat history and AI processing.
  4. Saga Pattern: Orchestrate multi-step workflows (message → AI → response → storage).

Benefits for chatbots:

  • Scalability: Independent scaling of chat, AI, and storage services.
  • Resilience: Queue messages during AI service downtime.
  • Real-time: Stream responses without blocking UI.
  • Auditability: Event sourcing provides complete conversation history.

Worked Example

Simplified event-driven chat flow:

Client → 1. Send message → Gateway Gateway → 2. Publish MessageSent → Kafka ChatService → 3. Subscribe MessageSent → Kafka ChatService → 4. Store message → DB ChatService → 5. Publish AIRequest → Kafka AIService → 6. Subscribe AIRequest → Kafka AIService → 7. Call LLM → LLMProvider AIService → 8. Publish AIResponse → Kafka ChatService → 9. Subscribe AIResponse → Kafka ChatService → 10. Stream to client → WebSocket

Kafka topics:

  • chat.message.sent - User messages
  • chat.ai.request - AI processing requests
  • chat.ai.response - AI streaming responses
  • chat.system.notification - Typing indicators, errors

Code sketch (Node.js with KafkaJS):

// Producer (Chat Service)
await producer.send({
  topic: 'chat.message.sent',
  messages: [{ value: JSON.stringify({ messageId, userId, content, timestamp }) }]
});

// Consumer (AI Service)
consumer.subscribe({ topic: 'chat.ai.request', fromBeginning: false });
consumer.run({
  eachMessage: async ({ message }) => {
    const request = JSON.parse(message.value);
    const stream = await llmProvider.generateStream(request.prompt);
    for await (const chunk of stream) {
      await producer.send({
        topic: 'chat.ai.response',
        messages: [{ value: JSON.stringify({ messageId: request.messageId, token: chunk }) }]
      });
    }
  }
});

Exercise

Design a Kafka-based event system for a chatbot that supports:

  1. Multiple concurrent users with session isolation.
  2. Streaming AI responses with backpressure handling.
  3. Fallback to secondary AI provider on failure.
  4. Rate limiting per user.

Define the topics, consumer groups, and error handling strategy.

Answer Key

Topics:

  • chat.message.sent.{userId} - Partitioned by user for ordering
  • chat.ai.request.{priority} - Separate topics for normal/high priority
  • chat.ai.response.{messageId} - Response streaming per message
  • chat.system.dead-letter - Failed messages for retry

Consumer groups:

  • chat-service-group - Handles message storage and routing
  • ai-service-group - Processes AI requests (multiple instances)
  • analytics-group - Logs all events for monitoring

Backpressure: Use Kafka consumer lag monitoring; scale AI service instances based on lag.

Fallback: Implement circuit breaker pattern; on AI provider failure, publish to fallback topic with different consumer.

Rate limiting: Use Redis sliding window; check before publishing to AI request topic.
