Lesson 1: High-Level Architecture of a Support Chatbot System

Goal: Understand the core components and communication patterns of a support chatbot system.

Concept Explanation

A support chatbot system typically consists of three main layers:

  1. Client Layer: The user interface (web, mobile, or desktop) that handles user interaction.
  2. Backend Services Layer: APIs, authentication, message routing, and business logic. Can be monolithic or microservices.
  3. AI/NLP Layer: Language models, retrieval-augmented generation (RAG), conversation memory, and response generation.

These layers communicate via:

  • HTTP/WebSocket between client and backend.
  • Event-driven messaging (Kafka, RabbitMQ) between backend services for scalability.
  • Internal APIs between microservices.

Worked Example

Consider a simplified architecture:

Backend Services ├── API Gateway (Auth, rate limiting) ├── Chat Service (Session management, message routing) ├── AI Service (LLM orchestration, RAG) ├── User Service (Profiles, preferences) └── Analytics Service (Logging, metrics) AI/NLP Components ├── LLM Provider (OpenAI, Ollama) ├── Vector Database (pgvector, Pinecone) └── Knowledge Base (docs, FAQs) Data Stores ├── SQL Database (PostgreSQL) └── Cache (Redis)

Data flow for a user message:

  1. User sends message via client interface.
  2. API Gateway authenticates and routes to Chat Service.
  3. Chat Service stores message, triggers AI Service.
  4. AI Service generates response using LLM, streams tokens back.
  5. Response flows back through the same path to client.

Exercise

Draw a simple diagram showing the communication paths between:

  • Client interface
  • Backend API Gateway
  • Chat Service
  • AI Service
  • Database

Identify which protocols you would use for each connection (HTTP, WebSocket, event queue, internal API).

Answer Key

Client ↔ API Gateway: HTTPS for commands, WebSocket for real-time streaming

API Gateway ↔ Chat Service: Internal REST/gRPC API

Chat Service ↔ AI Service: Event queue (Kafka) for async processing, or gRPC for sync

AI Service ↔ LLM Provider: HTTPS API calls

Chat Service ↔ Database: SQL connection for persistence

AI Service ↔ Vector DB: Database connection for similarity search

Key insight: Use WebSocket for real-time streaming of AI responses to maintain low latency.
